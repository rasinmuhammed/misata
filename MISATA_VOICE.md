# Misata Voice Guide

This document defines how Misata should sound in public docs, examples, UI copy, and feature naming.

## The Goal

Misata should feel:
- clear
- intelligent
- warm
- memorable
- slightly magical

Misata should never feel:
- confusing
- self-indulgent
- vague
- overbranded
- like a joke project

## The Writing Formula

Use this order:

1. Plain-English explanation
2. Feature name
3. Example
4. Why it matters

If a magical term appears before the plain-English meaning, the writing is probably too clever.

## Tone

Write like a naturally talented engineer who can also tell a story.

That means:
- short sentences when teaching
- richer sentences when framing the product
- no buzzword fog
- no empty hype

Good:

`Misata can shape a time-based story, then generate rows that actually add up to it.`

Bad:

`Misata unlocks a paradigm-shifting tapestry of synthetic intelligence for every possible data destiny.`

## Naming Rules

### Rule 1

The API stays obvious.

Examples:
- `SchemaConfig`
- `generate_with_reports`
- `outcome_curves`
- `workflow_preset`

### Rule 2

The docs can add a memorable layer.

Examples:
- `Time Machine` for temporal shaping
- `Spellbook` for reusable recipes
- `Oracle` for validation and reports

### Rule 3

Every magical term must map cleanly to a real feature.

If the term cannot be explained in one sentence, it is probably not good enough.

## Approved Style

This is the kind of phrasing we want:

- `Wave the Wand to get a first dataset quickly.`
- `Use the Time Machine when the story is about growth, seasonality, or timing.`
- `Save a working run in a Spellbook so you can replay it later.`
- `Ask the Oracle for validation and quality reports.`

## Not Approved

Avoid:
- fake mysticism
- overdesigned fantasy metaphors
- renaming obvious technical concepts into something obscure

Bad examples:
- `Summon the Chrono Sigil to awaken the entropy chamber`
- `ArcaneEntitySoul` as a class name
- `MysticOutputManifest` when `RunManifest` already works

## Documentation Rules

For every user-facing feature doc:
- state what it does in plain English
- state when to use it
- show one example
- mention the Misata term only if it helps memory

## Product Positioning

Misata is not trying to be cute.

Misata is trying to be:
- rigorous enough for engineers
- readable enough for normal users
- memorable enough to stand out

That means the magical language is seasoning, not the meal.
