# Data provenance

The questions procurement and security teams ask about synthetic data, each
answered with the mechanism that makes the answer true, not just a policy.
`misata provenance <dir>` prints the same statement plus per-file hashes for
any generated folder.

## Is any real data involved?

No. Misata generates from declarations: a schema, distributions, curves,
identities. There is no training step, no fitting to customer records, no
corpus of scraped personal data behind the values. Names, emails, and
addresses come from public-vocabulary pools and format templates, not from
people. The exception is explicit and opt-in: `misata mimic` fits marginal
distributions to a table you provide, on your machine, and stores only
quantiles, never rows.

## Does generation touch the network?

No. Generation is fully offline. The optional LLM integration operates only
at schema-design time (turning a sentence into a schema you then review), and
even that is opt-in with your own key. The generated bytes are produced by a
deterministic solver with no network access, no telemetry, and no phoning
home. There is no analytics or usage tracking anywhere in the library.

## Can you reproduce a dataset exactly?

Yes, to the byte. Same library version + same schema + same seed + same
generation mode = same bytes, on any machine, any OS. This is a tested
contract, not an aspiration: the suite asserts byte-identity across
round-trips, and in anchored mode a schema edit changes only the columns it
touches. A dataset is therefore fully specified by (version, schema file,
seed), which is three lines in an audit trail.

## Where do the realistic vocabularies come from?

Two sources, both license-clean:

- Curated built-in pools (city geodata, curriculum topics, product
  vocabulary) maintained in the repository under MIT.
- Wikidata-derived capsules, fetched at build time (never at generation
  time), carrying per-value provenance: the source QID and the CC0 license
  are recorded inside the capsule file itself. A capsule is a plain JSON
  file you can read, diff, and pin by hash.

Nothing is scraped from the open web at run time, and every capsule value
passed a validator that rejects placeholders, prose, and junk.

## What does the LLM see, when one is used at all?

Only your schema intent: the sentence you type and the table/column names it
implies. It never sees generated rows, never sees real data (there is none),
and never participates in generation. Its output is a schema document you
can read and edit before any data exists. Bring your own key; nothing is
proxied through Misata infrastructure.

## What guarantees does the output carry?

Declared properties are exact and machine-checkable: aggregate curves land
to the cent, ledgers balance, group shares hold, waterfalls and stock
ledgers reconcile, foreign keys have zero orphans. `misata audit` re-checks
the invariant catalog on the output, and an evalpack additionally re-executes
every shipped question with DuckDB, an engine that shares no code with the
generator. Where the engine cannot satisfy a declaration it refuses loudly;
the failure modes are documented, not hidden.
